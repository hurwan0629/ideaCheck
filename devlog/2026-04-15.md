## 오늘 결정한 것들

### 서비스 정체성
- 서비스명: **SYM (Search Your Market)**
- 슬로건: "당신의 아이디어, 시장에서 검증하세요"
- 핵심 가치: 창업자가 아이디어 구상 단계에서 시장/경쟁사 분석을 AI로 빠르게 할 수 있게

---

### 로그인 방식 — 강제 로그인 채택

**고민했던 것:**
게스트도 분석 기능 쓸 수 있게 할지 vs 로그인 필수로 할지

**결정: 로그인 필수 (간편 로그인)**

이유:
- 무료 사용자 토큰 소비 관리 필요
- 이력 저장, 재분석 루프 기능을 위해 어차피 계정 필요
- 이탈 가능성보다 관리 편의성을 택함
- 헤더 중앙 3개(트렌드/시장점유율/인기검색어)는 게스트 접근 허용

---

### 사용자 여정

```
[게스트]
  → 트렌드 / 시장분포 / 인기검색어 (로그인 불필요)
  → 아이디어 분석 클릭 → 로그인 요구

[로그인 후]
  → 아이디어 입력 (린 캔버스 형태, core_idea 필수 나머지 선택)
  → "AI 분석 시작하기" 클릭 → 즉시 처리
  → 결과: SWOT + 점수 5개 + 경쟁사 카드 + 차별화 전략
  → 채팅으로 조언 → 아이디어 수정 → 재분석 루프
  → 마이페이지에서 이력 관리
```

### 무료/유료 구분 방식

**고민했던 것:**
- 화면 자체를 다르게 보여주는 방식
- 같은 화면인데 일부 블러 처리하는 방식

**결정: 블러 처리 방식**

이유: 유료 전환 유도에 더 효과적. "이런 정보가 더 있어요"가 보여야 업그레이드 동기 생김.
MVP에서는 전부 오픈. 나중에 PLAN 테이블 기반으로 제한 추가.

### 채팅 AI — Claude API + RAG

**고민했던 것:**
GPT API 따로 쓸지 vs 자체 Claude API 쓸지

**결정: Claude API + RAG (DB 기반 컨텍스트)**

흐름:
```
사용자 질문
  → 관련 경쟁사/시장 데이터 DB에서 SELECT
  → [이전 대화 + DB 데이터 + 질문] → Claude 답변
```

비용 최적화:
- 최초 분석: Claude Sonnet (정확도 중요)
- 채팅 조언: Claude Haiku (비용 절감)

### DB 구조 주요 결정들

**USER_SUBSCRIPTIONS — 1:N 채택**

고민: 한 유저에 구독 하나니까 1:1 아닌가?
결정: 1:N. 이유는 구독 이력 보존 필요.
```
user_id=1, plan=FREE,    status=EXPIRED
user_id=1, plan=PREMIUM, status=ACTIVE
```
현재 구독은 `WHERE status='ACTIVE'`로 조회.

**POLICY_TYPES — POLICY_PROPS 컬럼 추가**

제안: POLICY_TYPES에 `policy_props` JSONB를 추가해서 해당 정책 유형에 맞는 JSONB 필드 목록을 정의.
```
policy_types.id=1, policy_props=["tier", "base_price", "enterprise_contact"]
```
크롤러/AI가 정책 수집 시 `policy_props`를 읽고 `COMPETITOR_POLICIES.POLICY_DATA`를 그 형태에 맞게 채움.

**COMPETITORS.DESCRIPTION / TARGET_CUSTOMER — RAG(벡터) 채택**

고민: 카테고리화 할지 vs 자유 텍스트 벡터로 저장할지
결정: pgvector로 벡터화. "B2B SaaS, 중소기업 세무" 같은 자유 텍스트가 유사도 검색에 더 적합.

**FEATURE_CATEGORIES / POLICY_TYPES — 시드 데이터로 해결**

nullable이라 "active인 게 하나도 없으면 크롤러가 분류 못 함" 문제 있음.
해결: 앱 최초 실행 시 기본 카테고리/정책 유형을 seed INSERT.
MVP에선 직접 DB에 넣고, 나중에 어드민 API로 관리.
