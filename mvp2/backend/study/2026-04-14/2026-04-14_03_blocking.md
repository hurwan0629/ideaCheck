## 0. 왜 알아야 하나

이벤트 루프 원리를 알았다면 블로킹이 왜 치명적인지 이제 정확히 이해할 수 있다.
"블로킹 = 스레드 점유"라는 막연한 이해에서 → "어떤 블로킹이고 어떤 해결책인가"로 넘어간다.

---

## 1. 블로킹의 정확한 정의 (재정립)

> **이벤트 루프 스레드가 다음 iteration으로 넘어가지 못하고 멈춰있는 상태**

01에서 본 루프를 기억하면:

```python
while True:                          # ← 여기가 계속 돌아야 함
    while ready_queue:
        callback = ready_queue.popleft()
        callback()   # ← 여기서 멈추면 루프 전체가 멈춤
    io_events = os_poll(timeout=...)
    ...
```

`callback()` 안에서 무언가 오래 걸리면 → 루프가 그 동안 멈춤 → 다른 요청 처리 불가.

---

## 2. I/O 블로킹 — await로 해결되는 이유

### 블로킹 상황

```python
# 동기 방식 (블로킹)
def get_data():
    response = requests.get("https://api.example.com")  # 1초 대기
    # 이 1초 동안 스레드가 아무것도 안 하고 기다림
    return response.json()
```

스레드가 OS에게 "이 소켓에서 데이터 줘" 하고 응답 올 때까지 그냥 기다린다.
기다리는 동안 다른 일을 못 한다.

### await가 해결하는 방법

```python
# 비동기 방식 (await)
async def get_data():
    response = await httpx_client.get("https://api.example.com")
    return response.json()
```

내부적으로:
```
1. httpx가 OS에게 "이 소켓 데이터 오면 알려줘" 등록 (epoll/kqueue)
2. Future 객체 생성
3. await → 코루틴이 Future를 yield → 이벤트 루프에 제어권 반환
4. 이벤트 루프: 다른 코루틴 실행 중...
5. OS: "소켓 준비됨" → 이벤트 루프가 future.set_result(data) 호출
6. 코루틴 재개 → response 받음
```

**핵심:** 기다리는 동안 스레드가 다른 코루틴을 처리한다.

## 3. CPU 블로킹 — await로 해결되지 않는 이유

```python
async def heavy_calc():
    # await가 없음
    result = sum(x**2 for x in range(100_000_000))  # 3초 걸리는 계산
    return result
```

이 경우:
```
이벤트 루프가 heavy_calc 실행 시작
  → sum(x**2 ...) 계산 중 (3초)
  → await가 없으므로 이벤트 루프에 제어권 반환 안 함
  → 3초 동안 루프 완전히 멈춤
  → 그 동안 HTTP 요청 전혀 처리 못함
```

**왜 await로 안 되나:**
- I/O 블로킹: OS에게 작업을 위임할 수 있음 (소켓 등록)
- CPU 블로킹: OS에게 위임할 수 없음. 계산 자체가 이 스레드에서 돌아야 함.
- await를 아무리 붙여도 계산 도중 이벤트 루프에 제어권을 줄 방법이 없음.

---

## 4. CPU 블로킹 해결책 — run_in_executor

별도 **스레드풀**에서 CPU 작업을 실행하고, 그 완료를 Future로 기다린다.

```python
import asyncio

async def good_heavy_calc():
    loop = asyncio.get_event_loop()
    result = await loop.run_in_executor(
        None,          # None = 기본 스레드풀 (ThreadPoolExecutor)
        heavy_calc,    # 실행할 동기 함수
        arg1, arg2     # 함수 인자
    )
    return result
```

내부 동작:
```
1. heavy_calc를 스레드풀의 워커 스레드에 제출
2. Future 생성
3. await → 이벤트 루프에 제어권 반환
4. 이벤트 루프: 다른 코루틴 처리
5. 워커 스레드에서 heavy_calc 완료 → future.set_result(result)
6. 이벤트 루프: future 완료 감지 → 코루틴 재개
```

**이벤트 루프 스레드는 계속 자유롭다.**

## 5. 세 가지 상황 비교

```
상황 1: I/O 작업 (httpx, Claude API, DB 쿼리)
  해결책: await
  이유: OS에게 위임 가능

상황 2: CPU 작업 (계산, 텍스트 처리, 이미지 변환)
  해결책: run_in_executor (스레드풀)
  이유: 계산을 다른 스레드로 분리

상황 3: 매우 무거운 CPU 작업 (ML 추론, 대용량 데이터 처리)
  해결책: ProcessPoolExecutor 또는 별도 프로세스
  이유: GIL 때문에 스레드풀로도 진짜 병렬 안 됨
```

```python
import asyncio
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

async def example():
    loop = asyncio.get_event_loop()

    # I/O: await
    data = await httpx_client.get(url)

    # 가벼운 CPU: run_in_executor (스레드풀)
    processed = await loop.run_in_executor(
        None,    # ThreadPoolExecutor (기본)
        parse_html, data.text
    )

    # 무거운 CPU: ProcessPoolExecutor
    with ProcessPoolExecutor() as pool:
        heavy_result = await loop.run_in_executor(
            pool,
            ml_inference, processed
        )
```

## 6. 숨겨진 블로킹 — 라이브러리 주의

겉으로 보면 평범한 코드인데 내부적으로 블로킹인 경우가 있다.

```python
async def sneaky_blocking():
    # (1) 동기 파일 읽기 — 블로킹
    with open("large_file.txt") as f:
        data = f.read()

    # (2) 동기 SQLAlchemy — I/O지만 블로킹
    result = db_session.execute(query)

    # (3) 동기 requests — 블로킹
    import requests
    resp = requests.get(url)  # httpx의 비동기 버전을 써야 함

    # (4) time.sleep() — 블로킹
    import time
    time.sleep(1)             # await asyncio.sleep(1) 써야 함
```

모두 `async def` 안에 있지만 전부 이벤트 루프를 블로킹한다.

## 이해도 측정

### 개념 문항

**C1.** I/O 블로킹은 await로 해결되고 CPU 블로킹은 왜 await로 해결이 안 되나요?
이벤트 루프 관점에서 설명해봐요.

```
답: 애초에 await 자체가 "cpu쓰지 않고 대기해야하는 직업은 나중에 와" 인데 cpu작업은 await 걸어도 결국 cpu를 사용해야하기 때문입니다.
```

**C2.** `run_in_executor`가 CPU 블로킹을 해결할 수 있는 이유가 뭔가요?

```
답: 별도의 스레드 (프로세스에게 할당된 cpu자원을 1/n 으로 나눠주기 때문에 cpu집중 작업에 맞는 지원)를 제공하기 때문입니다.
```

**C3.** 스레드풀(ThreadPoolExecutor)로도 GIL 때문에 CPU가 진짜 병렬 실행이 안 되는데,
그렇다면 CPU 작업에 run_in_executor를 쓰는 것의 의미가 뭔가요?

```
답: 이거가 잘 모르겠는데 I/O작업과 cpu작업 나누는건 알겠는데 가벼운 cpu블로킹 vs 무거운 cpu블로킹에서 run_in_executor과 ThreadPoolExecutor이 무슨 차이인지 모르겠습니다. 결국 cpu 자원 나누기 아닌가? 아니면 아예 프로세스 단위로 새로 생성해서 cpu레벨의 스케줄링을 통해 자원을 할당받는건가 싶습니다.
```

### 코드 감각 문항

**P1.** 아래 `async def` 함수들 중 이벤트 루프를 블로킹하는 것을 모두 골라봐요.

```python
# (a)
async def a():
    await asyncio.sleep(2)

# (b)
async def b():

    time.sleep(2)

# (c)
async def c():
    result = [i**2 for i in range(1000)]  # 빠른 계산

# (d)
async def d():
    result = [i**2 for i in range(100_000_000)]  # 느린 계산

# (e)
async def e():
    response = await httpx_client.get(url)
```

```
답: a: timezone에서 처리 (비동기임). b: await없어서 루프 callback에 가지도 않고 그냥 그대로 블로킹. c: 이정도는 b와 같이 바로 처리(하지만 빠름). d: c와 같지만 0.n~n초 걸림. e: httpx는 비동기 지원으로 알고있어서 비동기 됨. (아직은 뭐가 nonblocking 지원인지 외우진 못함)
```

```python
# 수정 후:
async def process_news(articles):
  # 답안: 하나하나 스레드를 만들게되면 스레드가 너무 많아질 수 있어서 텍스트 분석이 순서대로 진행되어도 된다는 가정하에 모든 작업을 하나의 별도 스레드에서 진행
```

## 학습 후 개념 교정 기록

### 몰랐던 것 → 이해한 것

---

**1. "이벤트 루프 해방"이 뭔지 몰랐다**

```
몰랐던 것:
  이벤트 루프 해방 = cpu 자원을 n분의 1로 나눠받는 것

이해한 것:
  해방 = 이벤트 루프 스레드가 CPU 계산에 묶이지 않는 것
  run_in_executor로 CPU 작업을 다른 스레드에 위임하면
  루프 스레드는 계속 I/O 이벤트/콜백을 처리할 수 있다.
  CPU가 2배 빨라지는 게 아니라, 루프가 멈추지 않는 게 목적.
```

---

**2. ThreadPoolExecutor와 ProcessPoolExecutor 구조를 반대로 알았다**

```
몰랐던 것:
  run_in_executor → 스레드 추가
  ThreadPoolExecutor → 프로세스 추가

이해한 것:
  run_in_executor(None 또는 ThreadPoolExecutor)
    → 같은 프로세스 내 스레드 추가
    → GIL 공유, 진짜 CPU 병렬 X, 루프 해방 O

  run_in_executor(ProcessPoolExecutor)
    → 별도 프로세스 생성
    → 각자 GIL, 진짜 CPU 병렬 O, 루프 해방 O

  [Python 프로세스]
  ├── 스레드1: 이벤트 루프
  ├── 스레드2: ThreadPoolExecutor 워커
  └── GIL: 한 번에 하나만 실행

  [별도 프로세스] ← ProcessPoolExecutor
  └── 자체 GIL, 독립적으로 CPU 사용
```

---

**3. CPU 점유 시간 분배 구조**

```
몰랐던 것:
  스레드도 결국 cpu 점유 시간을 나눠받는 거 아닌가?

이해한 것:
  맞음. 전체 구조는 규모만 다를 뿐 같은 개념.

  [OS 레벨]     프로세스1 → 프로세스2 → ...  (선점형, OS가 강제 전환)
  [스레드 레벨] 스레드1   → 스레드2   → ...  (선점형, OS가 강제 전환)
  [코루틴 레벨] 코루틴A   → 코루틴B   → ...  (협력형, 자발적 양보)

  핵심 차이:
  OS 스케줄링  → 선점형 (preemptive): OS가 강제로 뺏어감
  이벤트 루프  → 협력형 (cooperative): await로 스스로 양보해야 전환

  그래서 await 없이 CPU 계산을 돌리면
  루프가 제어권을 뺏어갈 수 없어 블로킹이 된다.
```

## 채점 결과

### C1 — I/O vs CPU 블로킹, await로 해결 안 되는 이유 | **65 / 100**

> "await 자체가 'cpu 쓰지 않고 대기해야 하는 작업은 나중에 와'인데 cpu 작업은 await 걸어도 결국 cpu를 사용해야 하기 때문"

방향은 맞음. 하지만 "이벤트 루프 관점에서"라는 조건에 맞는 설명이 빠짐.

정확한 설명:
- I/O 블로킹: OS에게 소켓 등록 후 제어권 반환 가능 → 루프가 다른 코루틴 처리
- CPU 블로킹: OS에게 위임 불가. 계산 도중 `await` 포인트가 없으므로 제어권 반환 자체가 불가능 → 루프가 멈춤

"await를 걸어도 cpu를 써야 한다"보다 "await 포인트가 없어서 제어권을 반환할 수 없다"가 핵심.

---

### C2 — run_in_executor가 CPU 블로킹을 해결하는 이유 | **60 / 100**

> "별도의 스레드(cpu 자원을 1/n으로 나눠주기)를 제공하기 때문"

"별도 스레드"는 맞음. "cpu 자원을 1/n으로 나눠준다"는 표현은 정확하지 않음.

정확한 설명:
- CPU 작업을 워커 스레드에 제출하고 Future를 생성
- `await`로 제어권을 반환 → **이벤트 루프 스레드가 자유로워짐**
- 워커 스레드에서 계산 완료 → `future.set_result()` → 루프가 코루틴 재개

핵심은 "cpu 자원 분배"가 아니라 **"이벤트 루프 스레드를 해방시킨다"** 는 것.

---

### C3 — ThreadPoolExecutor로도 GIL 때문에 병렬 안 되는데 의미가 뭔가? | **40 / 100**

> "잘 모르겠음. 프로세스 단위로 새로 생성해서 cpu 레벨 스케줄링인가?"

스스로 모른다고 표시한 게 정직하고 맞음. 마지막 추측(프로세스 단위)은 ProcessPoolExecutor 얘기인데 방향 맞음.

정확한 설명:
- ThreadPoolExecutor + run_in_executor의 목적은 **"진짜 CPU 병렬"이 아니라 "이벤트 루프 해방"**
- GIL 때문에 스레드가 동시에 CPU를 쓰지는 못하지만, 이벤트 루프 스레드가 블로킹되지 않으므로 다른 I/O 요청은 계속 처리 가능
- 진짜 CPU 병렬이 필요하면 → `ProcessPoolExecutor` (별도 프로세스 = GIL 우회 가능)

```
ThreadPoolExecutor: 이벤트 루프 해방 O, 진짜 CPU 병렬 X
ProcessPoolExecutor: 이벤트 루프 해방 O, 진짜 CPU 병렬 O
```

---

### P1 — 블로킹하는 코드 고르기 | **80 / 100**

> b, d가 블로킹. a, e는 비동기. c는 빠름.

거의 맞음. c에 대한 표현이 약간 모호함.

정리:
- a: `asyncio.sleep` → 비동기 ✓
- b: `time.sleep` → 이벤트 루프 블로킹 ✓
- c: 기술적으로는 블로킹이지만 수 ms → 실무상 허용. "블로킹이긴 한데 빠르다"가 정확한 표현.
- d: 3초 걸리는 계산 → 명백한 블로킹 ✓
- e: httpx 비동기 지원 → 비동기 ✓

---

### P2 — CPU 블로킹 코드 수정 | **70 / 100**

```python
def async_process(datas):
    for data in datas:
        score = analyze_sentiment(data["content"])
        results.append(score)   # ← 버그: results 초기화 없음
    return results

return await asyncio.get_event_loop().run_in_executor(
    None, async_process, articles
)
```

전체 루프를 하나의 동기 함수로 래핑해서 run_in_executor에 넘기는 접근은 유효함.

**버그:** `results = []` 초기화가 없어서 NameError 발생.

수정:
```python
def async_process(datas):
    results = []   # ← 여기 추가 필요
    for data in datas:
        score = analyze_sentiment(data["content"])
        results.append(score)
    return results
```

### P3 — 동기 SQLAlchemy 비동기로 감싸기 | **55 / 100**

```python
def add_data_and_commit(session, data):
    session.add(data)
    session.commit()

asyncio.get_event_loop().run_in_executor(
    None, add_data_and_commit, session, data
)
```

접근 자체는 맞음. 두 가지 누락:

1. **`await` 없음** — `run_in_executor`는 Future를 반환하므로 `await` 필수. 없으면 실행은 되지만 완료를 기다리지 않음.
2. **`async def save_async` 함수 구조 없음** — 문제에서 함수 안에 작성하라고 했음.

올바른 답:
```python
async def save_async(session, data):
    loop = asyncio.get_event_loop()
    await loop.run_in_executor(
        None,
        sync_save,   # 문제에서 주어진 동기 함수 그대로 사용
        session, data
    )
```
