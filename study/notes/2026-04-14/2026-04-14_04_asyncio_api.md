## 0. 왜 알아야 하나

순차 await만 알면 이런 손해가 생긴다:
- 서로 무관한 API 10개를 순서대로 호출 → 10배 느려짐
- 동시 실행하고 싶은데 방법을 모름
- 어떤 상황에 어떤 API를 쓸지 판단이 안 됨

## 1. 순차 await의 문제

```python
# 현재 trends_crawler.py 구조 (단순화)
for keyword in keywords:          # 키워드 10개
    google = await _fetch_google_trends(keyword)   # 1초
    naver  = await _fetch_naver_datalab(keyword)   # 1초
# 총 10 × 2 = 20초

# 개선 후 (gather 사용)
for keyword in keywords:
    google, naver = await asyncio.gather(
        _fetch_google_trends(keyword),
        _fetch_naver_datalab(keyword),
    )
# 총 10 × 1 = 10초 (Google/Naver를 동시 요청)
```

## 2. asyncio.gather() — "모두 동시에, 모두 기다려"

### 기본 동작

```python
results = await asyncio.gather(coro1(), coro2(), coro3())
# - coro1, coro2, coro3을 동시에 Task로 등록
# - 셋 다 완료될 때까지 대기
# - 결과는 입력 순서와 동일한 리스트로 반환
```

### 내부 동작

```
gather(A, B, C) 호출 시:
  1. A, B, C 각각을 Task로 만들어 이벤트 루프에 등록
  2. 내부적으로 "셋 다 완료되면 나를 재개해줘" 라고 Future 등록
  3. await → 루프에 제어권 반환
  4. 루프: A, B, C를 번갈아 실행
  5. 마지막 Task 완료 → gather 재개 → 결과 리스트 반환
```

### 에러 처리

```python
# 기본: 하나라도 에러 → 즉시 예외 (다른 Task는 취소 안 됨, 백그라운드에서 계속)
try:
    a, b = await asyncio.gather(task_a(), task_b())
except Exception as e:
    print(f"실패: {e}")

# return_exceptions=True: 에러를 예외 대신 결과값으로 반환
results = await asyncio.gather(
    task_a(), task_b(), task_c(),
    return_exceptions=True
)
for i, r in enumerate(results):
    if isinstance(r, Exception):
        print(f"Task {i} 실패: {r}")
    else:
        print(f"Task {i} 성공: {r}")
```

크롤러처럼 **일부 실패해도 나머지는 계속 처리**해야 하면 `return_exceptions=True` 필수.

### 리스트로 gather

```python
keywords = ["SaaS", "AI", "핀테크"]

# 코루틴 리스트 만들어서 언패킹
coros = [_fetch_google_trends(k) for k in keywords]
results = await asyncio.gather(*coros)  # * 로 언패킹
# results[0] = "SaaS" 결과, results[1] = "AI" 결과, ...
```

## 3. asyncio.create_task() — "등록하고 나는 계속 진행"

### gather와 차이

```python
# gather: 등록하고 여기서 대기
a, b = await asyncio.gather(fetch_a(), fetch_b())
print("둘 다 끝난 뒤에 여기 도달")

# create_task: 등록하고 바로 다음 줄
task_a = asyncio.create_task(fetch_a())  # 등록만 함
task_b = asyncio.create_task(fetch_b())  # 등록만 함
print("아직 fetch_a, fetch_b 실행 중일 수도 있음")

# 나중에 결과 필요할 때
result_a = await task_a
result_b = await task_b
```

### 언제 create_task를 쓰나

**1. 결과가 필요하기 전에 먼저 시작시키고 싶을 때**

```python
async def daily_job():
    # 큐 소비는 독립적이므로 미리 시작
    queue_task = asyncio.create_task(consume_reanalysis_queue())

    # 이 동안 크롤링도 진행
    raw_news = await crawl_news()
    await process_market_news(raw_news)
    await detect_policy_changes(raw_news)

    # 큐 작업 완료 확인
    await queue_task
```

**2. "불 지르고 잊기 (fire-and-forget)" — 결과 신경 안 쓸 때**

```python
# 로그 저장 같은 부수 작업 (실패해도 메인 플로우에 영향 없음)
asyncio.create_task(log_to_db(event))
# await 안 함 → 완료 여부 신경 안 씀
# 단, 에러가 발생해도 모름 → 중요한 작업에는 쓰지 말 것
```

## 4. 순서 의존 관계 판단 — 어떤 API를 쓸지 결정하는 법

```
A와 B 사이에 순서가 있나?
  YES → 순차 await
  NO  → gather 또는 create_task

A의 결과가 B의 입력인가?
  YES → 순차 await (A 완료 후 B 시작)
  NO  → 동시 가능

B를 "먼저 시작"시키고 나중에 결과가 필요한가?
  YES → create_task
  NO  → gather
```

### 이 프로젝트 daily_job 분석

```python
async def daily_job():
    raw_news = await crawl_news()           # raw_news 필요 → 먼저 완료해야 함

    await process_market_news(raw_news)     # raw_news 사용 → 순차
    await detect_policy_changes(raw_news)   # raw_news 사용 → 순차
    # process와 detect는 서로 무관 → gather 가능!

    await crawl_trends()                    # raw_news 불필요 → 위와 동시 가능
    await consume_reanalysis_queue()        # detect 이후여야 큐에 항목이 있음 → 순차
```

최적화 버전:
```python
async def daily_job():
    raw_news = await crawl_news()

    # process와 detect는 둘 다 raw_news 사용, 서로 독립 → gather
    await asyncio.gather(
        process_market_news(raw_news),
        detect_policy_changes(raw_news),
    )

    # trends는 raw_news 불필요, 위와 독립 → 사실 crawl_news 이후 어디서든 가능
    await asyncio.gather(
        crawl_trends(),
        consume_reanalysis_queue(),  # detect 완료 후이므로 큐 항목 있음
    )
```

## 5. asyncio.Queue — 생산자-소비자 패턴

이 프로젝트의 `reanalysis_queue`는 현재 단순 Python `set`.
규모가 커지면 asyncio.Queue로 대체하는 것이 적절하다.

```python
import asyncio

queue = asyncio.Queue(maxsize=100)  # maxsize: 백프레셔용 한계

# 생산자 (producer)
async def producer():
    for item in data:
        await queue.put(item)    # 가득 차면 빌 때까지 대기 (백프레셔)

# 소비자 (consumer)
async def consumer():
    while True:
        item = await queue.get()  # 비어있으면 항목 올 때까지 대기
        await process(item)
        queue.task_done()         # 처리 완료 표시

# 함께 실행
await asyncio.gather(producer(), consumer())
```

WebFlux의 Flux와 Backpressure 개념과 유사.
`maxsize`를 넘으면 `put()`이 블로킹 → 소비자 속도에 맞춰 생산자 조절.

### 코드 문항 (P1~P3): 51/70

**P1 — 50점**

시간 "5초" 계산은 맞는데 **코드가 잘못됨**.

```python
# 작성한 코드 문제
asyncio.create_task(fetch_c())  # Task 변수에 저장 안 함
...
c = await fetch_c()             # 새로운 fetch_c() 실행 → 최적화 효과 없음
```

올바른 코드:
```python
async def job():
    task_c = asyncio.create_task(fetch_c())  # 변수에 저장
    a = await fetch_a()    # 3초
    b = await fetch_b()    # 2초 (여기까지 5초, c는 이미 4초 진행 중)
    c = await task_c       # 1초만 추가 대기
    return a, b, c
# 총 5초

# 또는 더 간단하게
a, b, c = await asyncio.gather(fetch_a(), fetch_b(), fetch_c())
# 총 4초 (가장 긴 c 기준)
```

**P2 — 40점**

`await` 빠짐 + 결과 변수 바인딩 없음 → 실행하면 작동 안 함.

```python
# 작성한 코드 문제

asyncio.gather(...)   # google_score, naver_score 어디서 왔나?
```

올바른 코드:
```python
for topic_type, kw_list in keywords.items():
    for keyword in kw_list:
        google_score, naver_score = await asyncio.gather(
            _fetch_google_trends(keyword),
            _fetch_naver_datalab(keyword),

        await asyncio.gather(
            _save_trend(keyword, topic_type, today, google_score, "google_trends"),
            _save_trend(keyword, topic_type, today, naver_score, "naver_datalab"),
```

**P3 — 70점**

2번은 맞는데 1번 설명이 틀림. asyncio는 **단일 스레드** 라 스레드 점유 개념이 없음.

실제 핵심 문제:
1. Task 레퍼런스를 저장하지 않으면 **GC가 수거**할 수 있음 (Python 3.9+ 경고 발생)
2. `run_all()`이 반환되면 **Task 완료를 보장 못 함** (caller가 이벤트 루프를 닫으면 Task가 중간에 취소됨)
3. 예외/결과를 알 수 없음 (이건 맞게 잡음)
