# Claude 행동 지침

## 세션 시작 시 필수 순서

1. `mvp2/backend/my_level.md` 먼저 읽기 → 현재 수준 파악
2. `devlog/` 최신 파일 확인 → 마지막 결정 사항 파악
3. 오늘 작업/공부 목표 확인 후 시작

---

## 프로젝트 구조

```
SearchYourMarket/              ← 루트
  study/                       ← 공부 전용
    notes/                     ← 개념 공부 노트 (FastAPI, asyncio 등)
      session_log.md           ← 공부 세션 로그
    practice/                  ← 모르는 개념 직접 써보기
  mvp/                         ← 1차 바이브코딩 시도 (복잡해져서 중단)
  mvp2/                        ← 현재 진행 중
    backend/
      app/                     ← FastAPI 앱 코드
      my_level.md              ← 현재 기술 수준 현황판
      work.md                  ← 설계 대화 로그 전체 (날것)
  devlog/                      ← 포트폴리오용 개발일지
    overview.md                ← 전체 여정 요약
    YYYY-MM-DD.md              ← 날짜별 결정 + 고민 기록
  idea/                        ← 프로젝트 설계 및 기능
  prototype/                   ← UI 프로토타입 HTML
  trash/                       ← 폐기한 아이디어/파일
```

---

## 공부 진행 방식

- **개념 공부** ("~공부하자", "~알아보자" 등) → `study/notes/YYYY-MM-DD/` 폴더 안에 `YYYY-MM-DD_주제.md` 파일로 작성
  - 오늘 날짜 폴더가 이미 있으면 그 안에 새 파일 추가
- **실습** ("~직접 써보자", "~실습해보자" 등) → `study/practice/YYYY-MM-DD_주제/` 폴더 만들고 그 안에서 진행

---

## 이 프로젝트의 목표

- **공부 + 개발 병행**: 코드를 짤 때 왜 이렇게 설계했는지 설명 포함
- **이해하면서 진행**: 모르는 채로 넘어가지 않기
- **수준 업데이트**: 새로 이해한 개념은 세션 끝에 `mvp2/backend/my_level.md` 반영

---

## 설명 방식 규칙

- Spring/WebFlux 비유 적극 활용 (Uvicorn=톰캣, boundedElastic=스레드풀 등)
- CS 기초 (동시성/병렬성/스레드 개념)는 이미 이해함 → 설명 생략
- 모르는 API나 패턴은 짧은 예시 코드와 함께 설명
- 설계 이유("왜 이렇게 했는가")를 코드와 함께 항상 설명
- 모른다고 솔직하게 쓰는 스타일 → 아는 척 유도 금지

---

## 모르는 것 기록 방식

대화 중 설명했는데 모를 것 같아 보이는 개념이 나오면 `study/unknown_concepts.md`에 추가:
```
- 개념명: 한 줄 설명 (YYYY-MM-DD 발견)
```
- 이미 있는 항목은 중복 추가하지 말 것
- 나중에 공부할 때 참고하는 목록이므로 짧게 핵심만

## 세션 종료 시

### 공부 내용이 있었다면
`study/notes/session_log.md`에 로그 추가:
```
## YYYY-MM-DD | 주제
- 다룬 내용 요약
- 새로 이해한 것
- 아직 불명확한 것
```

`mvp2/backend/my_level.md`에서 변경된 수준이 있다면 해당 항목 업데이트.

### 설계 결정이 있었다면
`devlog/YYYY-MM-DD.md`에 개발일지 추가 또는 업데이트:
```
# YYYY-MM-DD | 주제

## 결정한 것
- 무엇을 → 왜 이렇게 결정했는지

## 고려했던 것들
- A 방식 vs B 방식 → 왜 B 선택

## 완료한 작업
- 실제로 한 것

## 다음 할 것
- [ ] 남은 것
```

> devlog는 포트폴리오용. 공부 내용 말고 설계 판단 + 트레이드오프만 담을 것.

### 문제 상황이 있었다면
`devlog/YYYY-MM-DD.md`의 `## 트러블슈팅` 섹션에 추가:
```
## 트러블슈팅
- **에러명/현상**: 원인 → 해결 방법
```
> 에러 자체보다 **왜 발생했는지 + 어떤 판단으로 해결했는지**를 담을 것.
