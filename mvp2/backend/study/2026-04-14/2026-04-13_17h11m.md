## Base 클래스의 역할
`db.py:17`에서 이렇게 정의됨:
```python
 from sqlalchemy.orm import DeclarativeBase
class Base(DeclarativeBase):
     pass
 ```
### DeclarativeBase가 하는 일
`DeclarativeBase`를 상속받은 순간 `Base`는 단순한 클래스가 아니라 **메타데이터 레지스트리**가 됨. 내부적으로 `Base.metadata`라는 객체를 가지게 되는데, 여기에 이 Base를 든 ORM 모       
        델들의 테이블 정보가 등록됨.
### 상속 흐름
```
 DeclarativeBase (SQLAlchemy 제공)
        ↓
     Base (db.py — 프로젝트 전용 레지스트리)
        ↓               ↓
   User (USERS)    Competitor (COMPETITORS)  ... 나머지 모든 모델
 ```
`user.py:28`의 `class User(Base):`와 `competitors.py:23`의 `class Competitor(Base):`처럼 모든 ORM 모델이 같은 `Base`를 상속받기 때문에 한 곳에서 묶여 관리됨.
### 핵심 이점 — `create_all`
`main.py`에서 아마 이런 코드가 있을 텐데:
```python
 Base.metadata.create_all(engine)
 ```
이게 가능한 이유가 바로 그거야. `Base`가 `User`, `Competitor`, 나머지 모든 테이블 클래스를 알고 있으니까, 이 한 줄로 DB에 모든 테이블을 한꺼번에 생성할 수 있음.
만약 `Base` 없이 각자 따로 관리했다면 테이블마다 개별로 DDL을 날려야 했겠지.
**한 줄 요약:** `Base`는 "이 프로젝트에서 ORM으로 관리할 테이블들의 등록부". 상속받는 것 = 그 등록부에 등록하는 것.

역할을 나눠서 비교하면:

| JPA (Spring)                        | SQLAlchemy (이 프로젝트)              |
|-------------------------------------|---------------------------------------|
| @Entity + @Table 애노테이션         | class User(Base): + __tablename__     |
| EntityManagerFactory (스키마 메타데이터 보유) | Base (= Base.metadata 레지스트리) |
| EntityManager / 영속성 컨텍스트     | Session (SessionLocal())              |

Base는 스키마 정의 + 등록부. "어떤 테이블이 어떤 컬럼을 갖는지"를 아는 것까지가 역할 끝.
런타임에 엔티티 상태를 추적하거나 관리하진 않음.

영속성 컨텍스트 역할을 하는 건 Session.
db.py:14, db.py:20의 SessionLocal / get_db()가 그 역할.

JPA 영속성 컨텍스트가 하는 것들:
- 1차 캐시 (같은 PK 조회 시 DB 재조회 안 함) → Session이 동일하게 함
- 변경 감지 (dirty checking) (commit 시 변경된 엔티티 자동 UPDATE) → Session이 동일하게 함
- 동일성 보장 (같은 세션 내 같은 PK = 같은 객체 인스턴스) → Session이 동일하게 함
- flush / commit → db.commit() / db.flush()

def get_db():
    db = SessionLocal()   # 영속성 컨텍스트 시작
    try:
        yield db          # 이 db 객체가 EntityManager 역할
    finally:
        db.close()        # 영속성 컨텍스트 종료

한 줄 요약:
- Base = JPA의 EntityManagerFactory 수준의 스키마 레지스트리
- Session = JPA의 EntityManager = 영속성 컨텍스트

[2026-04-13 13:35:30] - [답변]

### SessionLocal — Connection에 가까운 개념 맞음

| 개념               | SQLAlchemy           | JDBC/Spring                    |
|--------------------|----------------------|--------------------------------|
| DB 연결 풀 관리    | engine               | DataSource                     |
| 실제 연결 1개      | Session (SessionLocal()) | Connection + 영속성 컨텍스트 |
| 트랜잭션 단위      | Session 생명주기     | @Transactional 범위            |

### IoC / DI 복기 — 왜 쓰는가

DI 없이 짠다면:

@app.get("/users")
def get_users():
    db = SessionLocal()       # 핸들러가 직접 DB 연결 생성
    try:
        users = db.query(User).all()
        return users
    finally:
        db.close()            # 핸들러가 직접 정리도 책임

문제점:
- 모든 핸들러마다 SessionLocal() / try / finally / close() 반복
- 핸들러가 "DB 연결을 어떻게 만드는지"까지 알아야 함 (관심사 오염)
- 테스트할 때 DB 연결을 교체할 방법이 없음

### FastAPI의 Depends() — IoC 컨테이너 역할

# db.py — 연결 생성/해제 로직은 여기서만 관리
def get_db():
    db = SessionLocal()
    try:
        yield db          # 여기서 핸들러에게 db를 넘겨줌
    finally:
        db.close()        # 핸들러 끝나면 자동 정리

# router — 핸들러는 그냥 db를 "받기만" 함
@app.get("/users")
def get_users(db: Session = Depends(get_db)):
    return db.query(User).all()

Depends(get_db)가 하는 일:
  요청 들어옴 → FastAPI가 get_db() 실행해서 db 객체 생성
  → get_users(db=...) 호출 (핸들러는 받기만 함)
  → 핸들러 종료 → FastAPI가 get_db()의 finally 블록 실행 → db.close()

Spring의 @Autowired + @Transactional이 합쳐진 것과 비슷한 느낌.

### 왜 쓰는가 — 3가지 이유

1. 관심사 분리: 핸들러는 비즈니스 로직만. DB 연결 생성/해제는 get_db()만.
2. 재사용: get_db() 하나를 수십 개 라우터에서 Depends(get_db)로 재사용.
3. 테스트 교체 (가장 중요):
   app.dependency_overrides[get_db] = get_test_db
   핸들러 코드 건드리지 않고 의존성만 바꿔끼울 수 있음. Spring의 @Bean 오버라이딩과 같은 개념.

한 줄 요약: Depends()는 FastAPI의 IoC 컨테이너. 핸들러가 "어떻게 만드는지" 몰라도 되게 해주고,
생명주기 관리(close)까지 자동으로 해줌.

[2026-04-13 13:45:30] - [답변]

### 1. 트랜잭션 관리 — 가장 큰 차이

Spring은 @Transactional 하나면 commit/rollback 자동.
FastAPI + SQLAlchemy는 직접 써야 함:

@app.post("/users")
def create_user(user_data: UserCreate, db: Session = Depends(get_db)):
    try:
        user = User(**user_data.dict())
        db.add(user)
        db.commit()      # 직접 commit
        db.refresh(user) # commit 후 DB에서 갱신된 값 다시 읽어오기
        return user
    except Exception:
        db.rollback()    # 직접 rollback
        raise

get_db()의 finally: db.close()는 연결 반납만 해줄 뿐, commit/rollback은 안 해줌.

### 2. DI 컨테이너 무게 차이

Spring IoC 컨테이너: 앱 시작 시 빈 생성, 싱글톤 기본 (앱 종료까지 유지)
FastAPI Depends: 요청마다 실행됨 (요청 1개당 get_db() 1번 실행)

싱글톤처럼 쓰고 싶으면 @lru_cache나 app.state를 따로 써야 함.

### 3. Spring이 자동으로 해주는데 FastAPI는 직접 해야 하는 것

| 항목              | Spring              | FastAPI                          |
|-------------------|---------------------|----------------------------------|
| commit            | @Transactional 자동 | db.commit() 직접                 |
| rollback          | 예외 시 자동        | db.rollback() 직접               |
| 세션 반납         | 자동                | get_db()의 finally: db.close()   |
| refresh (갱신값)  | 자동                | db.refresh(obj) 직접             |
| 빈 스코프 관리    | @Scope 애노테이션   | Depends 함수 직접 구현           |

한 줄 요약: FastAPI DI는 생명주기 연결/반납은 자동이지만,
트랜잭션(commit/rollback/refresh)은 전부 개발자가 직접 관리해야 함.
Spring @Transactional 같은 자동화가 없음.

get_db()의 finally: db.close()로 반납은 보장되고, rollback은 예외 핸들러에서 패턴화해서 쓰면 돼.

[2026-04-13 13:55:30] - [답변]

SQLAlchemy는 크게 두 개의 레이어로 나뉨:

┌─────────────────────────────────────────┐
│           ORM Layer                     │  ← 이 프로젝트에서 쓰는 부분
│  (클래스 ↔ 테이블 매핑, Session 관리)   │
├─────────────────────────────────────────┤
│           Core Layer                    │  ← ORM 내부에서도 쓰고, 직접 쓸 수도 있음
│  (SQL 표현식, Engine, Connection)       │
├─────────────────────────────────────────┤
│           DBAPI (드라이버)              │  ← psycopg2, pymysql 등
└─────────────────────────────────────────┘

### Core Layer — engine
engine = create_engine(DATABASE_URL)
- DB와의 연결 풀을 관리하는 객체
- 실제 SQL을 DB에 보내는 통로
- Oracle/MySQL/PostgreSQL 방언(dialect) 차이를 여기서 흡수
  - URL만 바꾸면 나머지 코드는 그대로

### ORM Layer — 3가지 구성요소

① 모델 (Mapped Class): 테이블 정의
② Session: 작업 단위 (Unit of Work)
   - 변경된 객체들을 추적하다가 commit() 시점에 한꺼번에 SQL을 날림
③ Query / Select: 조회 방식
   - 구버전(1.x): db.query(User).filter(User.user_id == 1).first()
   - 신버전(2.x): select(User).where(User.user_id == 1) → db.execute(stmt).scalar_one()

### 이 프로젝트에서 앞으로 공부할 흐름
① 모델 정의 (Base, Mapped, mapped_column)   ← 완료
② 관계 설정 (relationship, ForeignKey)      ← 다음 단계
③ CRUD 쿼리 작성 (query / select)
④ 트랜잭션 관리 (commit, rollback, refresh)
⑤ 마이그레이션 (Alembic)

한 줄 요약: SQLAlchemy = Core(엔진/연결) + ORM(모델/세션/쿼리) 두 레이어.
이 프로젝트는 ORM 레이어를 쓰고, Core는 engine으로만 노출돼 있음.

### 1. ORM 제약조건 → DB 반영 방식
create_all() 시점에 DDL로 변환돼서 실제 DB에 걸림.

- BigInteger → BIGINT
- ForeignKey("USERS.USER_ID") → FOREIGN KEY ... REFERENCES USERS(USER_ID)
- nullable=False → NOT NULL
- unique=True → UNIQUE 제약조건

### Enum — 두 가지 레이어에서 동시에 동작
- Python Enum: Python 코드에서 타입 안전성
- SqlEnum(AuthProvider): DB에 ENUM('LOCAL','GOOGLE','KAKAO') 제약조건으로 생성
- str, Enum으로 상속하는 이유: JSON 직렬화 시 "LOCAL" 문자열로 바로 나오게 하기 위함

### default vs DB default
- default=lambda: datetime.now(timezone.utc) → Python이 INSERT 직전에 채움
- server_default="now()" → DB 서버가 INSERT 시 채움
현재 프로젝트는 Python 레벨 default 사용.

### 2. 컬럼/테이블 생성 규칙
- Mapped[str]        → NOT NULL
- Mapped[str | None] → NULL 허용
- mapped_column("DB컬럼명", 타입, 제약조건들...)

자주 쓰는 타입:
BigInteger → BIGINT, String(n) → VARCHAR(n), Text → TEXT
Boolean → BOOLEAN, DateTime(timezone=True) → TIMESTAMPTZ
Date → DATE, JSONB (pg 전용) → JSONB (JSON + 인덱싱)

### 3. 관계 맺는 방법
- ForeignKey: DB에 외래키 제약조건 생성
- relationship: Python 객체 간 탐색 가능하게 해줌
둘 다 선언해야 완전함. ForeignKey만 있으면 DB 무결성은 보장되지만 Python에서 user.auth_accounts 같이 접근 불가.

# user.py에 추가
auth_accounts: Mapped[list["AuthAccount"]] = relationship("AuthAccount", back_populates="user")

# auth_accounts.py에 추가
user: Mapped["User"] = relationship("User", back_populates="auth_accounts")

### lazy loading 주의
relationship 기본값은 lazy loading → 접근 시점에 SELECT 발생 → N+1 문제 가능.
해결: selectinload or joinedload로 미리 로딩.
users = db.query(User).options(selectinload(User.auth_accounts)).all()

요약:
1. create_all()시 DDL로 변환돼 DB에 직접 걸림. Enum은 Python + DB 이중으로 걸림
2. Mapped[타입] + mapped_column("DB컬럼명", 타입, 제약조건) 패턴
3. ForeignKey = DB 제약조건, relationship = Python 객체 탐색. 둘 다 선언해야 완전함

[2026-04-13 14:20:00] - [질문] 관계 형성하는거는 이미 ORM모델 생성시에 하는거 아니야? ForeignKey설정 하잖아. 왜 또 관계 형성을 하는거야? 그리고 컬럼단위에서 하는거야 아니면 User클래스 맨 아래에 코드 한줄 더 쓰는거야?

[2026-04-13 14:20:30] - [답변]

ForeignKey와 relationship은 역할이 완전히 다름.

ForeignKey → DB한테 하는 말. PostgreSQL에 FOREIGN KEY DDL이 걸림. DB가 무결성 체크.
relationship → Python한테 하는 말. DB랑 아무 상관 없음. Python 객체 탐색 경로만 생김.

relationship 없을 때:
  user = db.query(User).filter(User.user_id == 1).first()
  accounts = db.query(AuthAccount).filter(AuthAccount.user_id == 1).all()  # 직접 쿼리 써야 함

relationship 있을 때:
  user = db.query(User).first()
  accounts = user.auth_accounts   # 속성처럼 접근 가능

위치: 클래스 안, 컬럼 선언 아래에 모아서 씀. DB에 컬럼이 생기지는 않음.

class User(Base):
    __tablename__ = "USERS"
    user_id: Mapped[int] = mapped_column(...)    # 실제 DB 컬럼
    user_name: Mapped[str] = mapped_column(...)  # 실제 DB 컬럼

    # 아래는 DB 컬럼 아님 — Python 탐색 경로
    auth_accounts: Mapped[list["AuthAccount"]] = relationship("AuthAccount", back_populates="user")
    user_ideas: Mapped[list["UserIdea"]] = relationship("UserIdea", back_populates="user")

한 줄 요약: ForeignKey는 DB 제약조건, relationship은 Python 객체 탐색.
완전히 다른 레이어의 설정이라 둘 다 필요함.

[2026-04-13 14:30:00] - [질문] enum, date(time), sqlalchemy제공 타입 이런것들은 그냥 보이는그대로 쓰면 되는거야? 모르면 나중에 문제 생기는거 당장은 딱히 없나?

[2026-04-13 14:30:30] - [답변]

대부분은 보이는 그대로 써도 되지만, 실제로 문제 터지는 함정 몇 개 있음.

### 1. DateTime(timezone=True) 함정
올바른 패턴: default=lambda: datetime.now(timezone.utc)
위험한 패턴: datetime.now() 또는 datetime.utcnow (timezone 정보 없는 naive datetime)
→ naive datetime은 PostgreSQL이 로컬 시스템 타임존으로 해석. 배포 환경 바뀌면 시간 데이터 꼬임.
현재 프로젝트는 timezone.utc로 잘 쓰고 있으니 이 패턴 유지.

### 2. Mapped[str | None] vs nullable — 반드시 일치해야 함
Mapped[str]        → nullable=False  (둘 다 NOT NULL)
Mapped[str | None] → nullable=True   (둘 다 NULL 허용)
Python 타입힌트와 DB 제약은 자동 동기화 안 됨. 손으로 맞춰야 함.

### 3. Enum — str, Enum 상속 안 하면 API 응답에서 터짐
class AuthProvider(Enum): → JSON 직렬화 시 "AuthProvider.LOCAL" 또는 에러
class AuthProvider(str, Enum): → JSON 직렬화 시 "LOCAL"  ← 올바른 패턴
새 Enum 추가 시 str, Enum 상속 패턴 유지 필수.

### 4. BigInteger PK — auto increment 안 됨
BigInteger + primary_key=True만으로는 PostgreSQL에서 시퀀스 자동 생성 안 됨.
INSERT 시 user_id를 직접 넣거나, Sequence("users_user_id_seq") 추가 필요.
또는 Integer + primary_key=True만 써도 PostgreSQL이 자동으로 SERIAL 처리.

### 5. JSONB — PostgreSQL 전용
from sqlalchemy.dialects.postgresql import JSONB
PostgreSQL에서만 동작. 테스트 환경에서 SQLite 쓰면 깨짐.

| 타입/패턴          | 주의사항                                              |
|--------------------|-------------------------------------------------------|
| DateTime           | datetime.now(timezone.utc) 패턴 유지                  |
| Mapped + nullable  | 반드시 쌍으로 일치                                    |
| Enum               | str, Enum 상속 필수                                   |
| BigInteger PK      | auto increment 안 됨, Sequence 추가 필요              |
| JSONB              | PostgreSQL 전용, SQLite 테스트 불가                   |
