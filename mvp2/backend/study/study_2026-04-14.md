## 2. APScheduler — 자동 실행 스케줄러

**설치:** `pip install APScheduler`
**공식문서:** https://apscheduler.readthedocs.io/en/3.x/

### 무슨 라이브러리인가?
특정 시각이나 일정 간격으로 **파이썬 함수를 자동 실행**시키는 라이브러리.
Linux의 cron, Windows의 작업 스케줄러를 파이썬 코드 안에서 처리.

### 이 프로젝트에서 쓰는 이유
- FastAPI 앱이 켜져 있는 동안 **매일 오전 3시에 크롤링**을 자동으로 돌려야 하기 때문
- FastAPI는 `async` 기반이라 → `AsyncIOScheduler` (async 호환 버전) 사용

### 핵심 개념

| 개념 | 설명 |
|------|------|
| `AsyncIOScheduler` | FastAPI(asyncio) 환경과 호환되는 스케줄러 |
| `CronTrigger` | cron 문법으로 실행 시각 지정 (매일 3시 등) |
| `IntervalTrigger` | N초/분/시간마다 반복 실행 (현재 프로젝트에서 import만 해둠) |
| `add_job()` | 함수 + 트리거 등록 |
| `replace_existing=True` | 앱 재시작 시 같은 id 잡 중복 등록 방지 |

### 프로젝트 코드 연결 (`scheduler.py`)

```python
from apscheduler.schedulers.asyncio import AsyncIOScheduler
from apscheduler.triggers.cron import CronTrigger

scheduler = AsyncIOScheduler()

def setup_scheduler():
    # 매일 03:00 실행
    scheduler.add_job(
        daily_job,
        CronTrigger(hour=3, minute=0),
        id="daily_job",
        replace_existing=True,
    )

    # 1, 4, 7, 10월 1일 04:00 실행 = 분기 1회
    scheduler.add_job(
        quarterly_job,
        CronTrigger(month="1,4,7,10", day=1, hour=4, minute=0),
        id="quarterly_job",
        replace_existing=True,
    )
```

### FastAPI 연결 방법 (main.py에서)
```python
from contextlib import asynccontextmanager
from fastapi import FastAPI
from app.collector.scheduler import scheduler, setup_scheduler

@asynccontextmanager
async def lifespan(app: FastAPI):
    setup_scheduler()
    scheduler.start()   # 앱 시작 시 스케줄러 가동
    yield
    scheduler.shutdown()  # 앱 종료 시 스케줄러 중지

app = FastAPI(lifespan=lifespan)
```

### CronTrigger 문법 치트시트
```python
CronTrigger(hour=3, minute=0)                    # 매일 03:00
CronTrigger(day_of_week="mon", hour=9)           # 매주 월요일 09:00
CronTrigger(month="1,4,7,10", day=1, hour=4)    # 분기 1회
CronTrigger(minute="*/30")                       # 30분마다
```

## 3. httpx — 비동기 HTTP 클라이언트

**설치:** `pip install httpx`
**공식문서:** https://www.python-httpx.org/

### 무슨 라이브러리인가?
파이썬의 `requests` 라이브러리와 거의 동일한 API지만, **async(비동기) 지원**이 핵심 차이.

### requests vs httpx

| | requests | httpx |
|---|---|---|
| async 지원 | X | O |
| 문법 | `requests.get(url)` | `await client.get(url)` |
| FastAPI에서 사용 | 동작하지만 블로킹 | 권장 방식 |

### 왜 async가 중요한가?
```
동기 방식: URL 1 요청 → 응답 기다림 → URL 2 요청 → 응답 기다림 → ...
           뉴스 100개 × 1초 = 100초

비동기:   URL 1,2,3...100 동시 요청 → 오는 순서대로 처리
           대기 시간이 겹쳐서 훨씬 빠름
```

### 프로젝트 코드 연결 (`news_crawler.py`, `competitor_crawler.py`)

```python
import httpx

# AsyncClient: async with 블록 안에서만 사용. 블록 끝나면 연결 자동 해제.
async with httpx.AsyncClient(timeout=10.0) as client:
    response = await client.get(url)
    response.raise_for_status()   # 4xx/5xx면 예외 발생
    html = response.text          # 응답 HTML 문자열
```

### 주요 메서드

```python
# GET 요청
response = await client.get(url, params={"key": "value"})

# POST 요청 (JSON body)
response = await client.post(url, json={"key": "value"}, headers={"Authorization": "Bearer ..."})

# 응답 처리
response.text          # HTML/텍스트
response.json()        # JSON 파싱
response.status_code   # 200, 404 등
response.raise_for_status()  # 에러 코드면 예외 던짐
```

## 4. BeautifulSoup4 — HTML 파싱

**설치:** `pip install beautifulsoup4`
**공식문서:** https://www.crummy.com/software/BeautifulSoup/bs4/doc/

### 무슨 라이브러리인가?
HTML/XML 문자열에서 **원하는 태그나 텍스트를 꺼내는** 라이브러리.
httpx로 HTML을 받아온 뒤, 이걸로 원하는 부분만 추출.

### 프로젝트 코드 연결 (`news_crawler.py`, `competitor_crawler.py`)

```python
from bs4 import BeautifulSoup

# HTML 문자열 → BeautifulSoup 객체 (파서 지정 필요)
soup = BeautifulSoup(response.text, "html.parser")

# 태그 찾기
article = soup.find("article")                          # 첫 번째 <article> 태그
body = soup.find("div", class_="article-body")          # class 지정
title = soup.select_one("meta[name='description']")     # CSS selector
items = soup.select(".feature-item")                    # 여러 개 (리스트 반환)

# 텍스트/속성 꺼내기
text = tag.get_text(strip=True)                         # 태그 안 텍스트
attr_val = tag.get("content", "")                       # 특정 속성 값
```

### 파서 종류

| 파서 | 설치 | 속도 | 용도 |
|------|------|------|------|
| `html.parser` | 내장 (설치 불필요) | 보통 | 일반적인 경우 |
| `lxml` | `pip install lxml` | 빠름 | 속도가 중요할 때 |
| `html5lib` | `pip install html5lib` | 느림 | 깨진 HTML 처리 |

### 이 프로젝트에서 httpx + BS4 쓰는 경우
- 정적 HTML 사이트 (JS 없이 서버에서 바로 HTML 반환)
- 속도가 빠르고 리소스 소모가 적음
- JS 렌더링이 필요한 사이트는 → Playwright로 넘어감

## 5. Playwright — 브라우저 자동화

**설치:** `pip install playwright && playwright install chromium`
**공식문서:** https://playwright.dev/python/docs/intro

### 무슨 라이브러리인가?
실제 Chrome/Firefox 브라우저를 **코드로 조종**하는 라이브러리.
"버튼 클릭", "스크롤", "로그인", "JS 실행 후 HTML 캡처" 등 자동화 가능.

### httpx+BS4 vs Playwright 선택 기준

| 상황 | 선택 |
|------|------|
| 서버에서 HTML 바로 반환 (정적 사이트) | httpx + BeautifulSoup (빠름) |
| React/Vue 등 JS로 렌더링하는 사이트 | Playwright (느리지만 필수) |
| 로그인 필요한 사이트 | Playwright |
| 버튼 클릭 후 데이터 로딩 | Playwright |

### 왜 JS 사이트는 httpx로 안 되나?
```
httpx 요청 → 서버에서 빈 HTML 반환 (<div id="root"></div>만 있음)
             JS는 브라우저가 실행해야 내용이 채워짐
             → httpx만으로는 빈 페이지만 보임

Playwright → 실제 Chrome 실행 → JS 로딩 완료 대기 → 채워진 HTML 캡처
```

### 프로젝트 코드 연결 (`competitor_crawler.py`)

```python
from playwright.async_api import async_playwright

async def _crawl_with_playwright(url: str) -> dict:
    async with async_playwright() as p:
        # headless=True: 창 없이 백그라운드 실행 (서버 환경 필수)
        browser = await p.chromium.launch(headless=True)
        page = await browser.new_page()

        await page.goto(url)
        # networkidle: 네트워크 요청 0개 = JS 로딩 완료 신호
        await page.wait_for_load_state("networkidle")

        html = await page.content()  # JS 실행 후 최종 HTML
        await browser.close()

    soup = BeautifulSoup(html, "html.parser")
    # 이후 BS4로 파싱
```

### wait_for_load_state 옵션

| 옵션 | 설명 |
|------|------|
| `"load"` | 기본 페이지 로드 완료 |
| `"domcontentloaded"` | DOM 구성 완료 (JS 실행 전) |
| `"networkidle"` | 네트워크 요청 0개 (JS 완전 로딩) |

### 기타 자주 쓰는 Playwright 메서드
```python
await page.click("button.submit")            # 버튼 클릭
await page.fill("input[name='email']", "...")  # 입력창에 타이핑
await page.wait_for_selector(".data-table")  # 특정 요소 나타날 때까지 대기
text = await page.inner_text(".price")       # 텍스트 추출
screenshot = await page.screenshot()        # 화면 캡처
```

## 6. anthropic (Claude API) — AI 분석

**설치:** `pip install anthropic`
**공식문서:** https://docs.anthropic.com/en/api/getting-started

### 무슨 라이브러리인가?
Anthropic의 **Claude AI에게 텍스트를 보내고 답변을 받는** API 클라이언트.
"이 뉴스를 분류해줘", "경쟁사 강점/약점 분석해줘" 같은 작업을 코드로 자동화.

### 환경변수 설정
```bash

```
`anthropic.AsyncAnthropic()` 생성 시 자동으로 `ANTHROPIC_API_KEY` 환경변수를 읽음.

### 프로젝트 코드 연결

#### 기본 사용법 (`market_processor.py`, `policy_detector.py`, `analysis_generator.py`)
```python
import anthropic

claude = anthropic.AsyncAnthropic()  # async 버전

response = await claude.messages.create(
    model="claude-sonnet-4-6",       # 사용할 모델
    max_tokens=512,                  # 응답 최대 길이
    messages=[{"role": "user", "content": "질문 내용"}],

```

### 이 프로젝트에서 쓰는 모델별 역할

| 모델 | 용도 | 이유 |

| `claude-sonnet-4-6` | 경쟁사 종합 분석, 뉴스 분류 | 더 정확한 분석 필요 |
| `claude-haiku-4-5-20251001` | 정책 변경 감지 (뉴스 분류) | 빠르고 저렴 + 단순 분류 작업 |

## 7. pgvector — 벡터 유사도 검색 (RAG)

**설치:** `pip install pgvector`
**공식문서:** https://github.com/pgvector/pgvector-python
**PostgreSQL 확장 설치:** `CREATE EXTENSION vector;` (DB에서 실행)

### 무슨 라이브러리인가?
PostgreSQL에 **벡터 검색 기능을 추가**하는 확장.
텍스트를 숫자 벡터(임베딩)로 변환해두고, 의미가 비슷한 텍스트를 빠르게 검색.

### 왜 필요한가? (RAG 개념)

**문제:**
```
아이디어 분석 시 DB의 경쟁사 300개 정보를 전부 Claude에 넘기면
→ 토큰 폭발 (비용) + 컨텍스트 한계 초과
```

**해결 (RAG):**
```
1. [사전 작업] 각 경쟁사 정보 → 임베딩(숫자 벡터) → pgvector에 저장
2. [분석 시]   아이디어 입력 → 임베딩 → pgvector에서 "비슷한 경쟁사 Top-5"만 검색
3.             Top-5만 Claude에 전달 → 토큰 절감 + 더 정확한 분석
```

### 임베딩이란?
```
"세무 자동화 앱" → [0.12, -0.45, 0.89, ...]  (1536차원 숫자 배열)
"자동 세금 신고" → [0.11, -0.44, 0.91, ...]  (비슷한 의미 = 벡터 위치 가까움)
"강아지 산책" →   [-0.78, 0.23, -0.12, ...]  (관계없는 의미 = 벡터 위치 멀음)

코사인 거리로 "얼마나 비슷한가"를 0~1 수치로 계산 가능
```

### 프로젝트 코드 연결 (`analysis_generator.py`)

```python
from pgvector.sqlalchemy import Vector  # SQLAlchemy 컬럼 타입

# 모델에서 벡터 컬럼 정의 (예시)
from sqlalchemy.orm import Mapped, mapped_column
class CompetitorEmbedding(Base):
    embedding: Mapped[list[float]] = mapped_column(Vector(1536))

# 임베딩 생성 (OpenAI API 예시 — 현재 더미로 구현됨)
async def _create_embedding(text: str) -> list[float]:
    # 실제 구현 예정:
    # import openai
    # client = openai.AsyncOpenAI()
    # response = await client.embeddings.create(
    #     model="text-embedding-3-small",
    #     input=text,
    # )
    # return response.data[0].embedding
    return [0.0] * 1536  # 현재 더미
```

### pgvector 유사도 검색 쿼리
```sql
-- <=> 연산자: 코사인 거리 (0에 가까울수록 비슷)
SELECT competitor_id, embedding <=> :query_vector AS distance
FROM competitor_embeddings
ORDER BY distance
LIMIT 5;

-- <-> 연산자: L2 거리 (유클리드)
-- <#> 연산자: 내적 (dot product)
```

### 임베딩 텍스트 구성 방법 (`analysis_generator.py`)
```python
def _build_embedding_text(context: dict, analysis: dict) -> str:
    parts = [
        context.get("description", ""),
        f"타겟 고객: {context.get('target_customer', '')}",
        f"강점: {', '.join(analysis.get('strength', []))}",
        f"약점: {', '.join(analysis.get('weakness', []))}",
        f"키워드: {', '.join(analysis.get('characteristic', {}).get('keywords', []))}",
    ]
    return " | ".join(filter(None, parts))
    # 결과: "금융 슈퍼앱 | 타겟 고객: MZ세대 | 강점: 간편 송금, 투자 | ..."
```

## 8. Python 내장: hashlib / set / async-await

### hashlib — URL 중복 체크

**파일:** `news_crawler.py`, `policy_detector.py`

```python
import hashlib

# URL → 고정 길이 해시 문자열 (중복 체크용)
url_hash = hashlib.md5(url.encode()).hexdigest()
# "https://example.com/article/123" → "d41d8cd98f00b204e9800998ecf8427e"

# JSON 데이터 해시 (내용이 바뀌었는지 체크)
data_hash = hashlib.md5(
    json.dumps(data, sort_keys=True).encode()
).hexdigest()
# sort_keys=True: 키 순서 상관없이 같은 내용이면 같은 해시
```

**왜 쓰나:** URL 자체를 저장하면 문자열 비교. 해시로 변환하면 고정 길이(32자)로 빠른 비교 가능.

---

### set — 재분석 큐 / 중복 방지

**파일:** `reanalysis_queue.py`, `news_crawler.py`

```python
# set: 중복 없는 집합. 같은 값이 여러 번 들어와도 1개만 유지.
reanalysis_queue: set[int] = set()

reanalysis_queue.add(competitor_id)   # 추가 (중복 무시)
reanalysis_queue.clear()              # 전체 비우기
targets = list(reanalysis_queue)      # 리스트로 변환

# in 연산자: O(1) 빠른 존재 여부 확인
if url_hash in _collected_url_hashes:
    continue
```

**리스트 vs set 선택 기준:**
- `list`: 순서 중요, 중복 허용
- `set`: 순서 무관, 중복 제거, `in` 연산이 빠름 (O(1) vs list O(n))

### async / await 패턴

**이 프로젝트 전체에서 쓰는 핵심 문법**

```python
# async def: 비동기 함수 선언
async def crawl_news() -> list[dict]:
    # await: 비동기 작업이 끝날 때까지 기다림 (다른 작업은 계속 진행)
    async with httpx.AsyncClient() as client:
        response = await client.get(url)
    return results

# async with: 비동기 컨텍스트 매니저 (자원 자동 해제)
async with httpx.AsyncClient() as client:
    ...

async with async_playwright() as p:
    ...
```

**주의:** `async def` 함수는 반드시 `await`로 호출해야 함. 그냥 호출하면 실행 안 됨.
```python
await crawl_news()   # 올바름
crawl_news()         # 코루틴 객체만 반환, 실제 실행 안 됨
```

## 9. 라이브러리 선택 기준 요약

| 상황 | 사용 라이브러리 | 이유 |
|------|----------------|------|
| 정적 HTML 사이트 크롤링 | httpx + BeautifulSoup | 빠름, 리소스 적음 |
| JS 렌더링 필요한 사이트 | Playwright | 실제 브라우저 실행 필요 |
| 뉴스/경쟁사 정보 분류 | Claude API (anthropic) | 규칙 기반으로 분류 불가 |
| 의미 기반 검색 | pgvector | 텍스트 유사도로 Top-K 검색 |
| 주기적 자동 실행 | APScheduler | FastAPI 내장 기능 없음 |
| URL/데이터 중복 체크 | hashlib | 고정 길이 해시로 빠른 비교 |
| 재분석 대기열 | Python set | 중복 제거 + 빠른 조회 |
