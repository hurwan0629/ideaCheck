# 04. get_db() 패턴 — Depends + SessionLocal의 결합

---

## 전체 구성

```python
# database.py
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker

engine = create_engine("postgresql://user:password@localhost/dbname")
SessionLocal = sessionmaker(bind=engine, autocommit=False, autoflush=False)

def get_db():
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
```

```python
# router.py
from fastapi import Depends
from sqlalchemy.orm import Session
from .database import get_db

@app.get("/items")
def read_items(db: Session = Depends(get_db)):
    return db.query(Item).all()
```

---

## 요청 하나가 들어왔을 때 전체 흐름

```
HTTP GET /items 요청
    ↓
FastAPI: read_items 파라미터 분석
    ↓
Depends(get_db) 발견 → get_db() 실행 시작
    ↓
db = SessionLocal()   ← 커넥션 풀에서 커넥션 가져옴
    ↓
yield db              ← db를 read_items(db=...) 에 주입
    ↓
read_items 실행 → db.query(Item).all()
    ↓
응답 반환
    ↓
finally: db.close()   ← 커넥션 풀에 반납
```

요청마다 세션이 새로 생기고, 응답 후 반드시 닫힌다.
세션 누수 없음.

---

## yield가 없으면 어떻게 되나

```python
# 잘못된 방식
def get_db_bad():
    db = SessionLocal()
    return db           # 반환하고 끝. close() 호출 불가.

@app.get("/items")
def read_items(db: Session = Depends(get_db_bad)):
    items = db.query(Item).all()
    # db.close() 누가 해주나? 아무도 안 함.
    return items
```

- `return`을 쓰면 FastAPI가 teardown(close) 시점을 알 방법이 없음
- 커넥션이 풀로 안 돌아가고 요청마다 하나씩 새로 열림
- 풀 고갈 → 새 요청이 커넥션 못 얻음 → 타임아웃

---

## 여러 Depends가 같은 get_db를 쓸 때

```python
def get_user_service(db: Session = Depends(get_db)):
    return UserService(db)

def get_order_service(db: Session = Depends(get_db)):
    return OrderService(db)

@app.post("/checkout")
def checkout(
    user_svc = Depends(get_user_service),
    order_svc = Depends(get_order_service),
):
    ...
```

FastAPI는 같은 Depends를 요청 내에서 **한 번만 실행**하고 캐싱한다.
`get_db()`가 두 곳에 쓰여도 세션은 하나만 생성됨.
→ 같은 트랜잭션 안에서 user_svc, order_svc가 동작.

---

## collector에서 Depends 없이 재현하기

FastAPI 라우터 밖(collector job)에서는 `Depends`를 쓸 수 없다.
`get_db()`의 내부 로직을 직접 따라하면 된다.

```python
# jobs/daily.py

from app.database import SessionLocal

def daily_job():
    db = SessionLocal()
    try:
        run_news_job(db)
        run_policy_job(db)
        run_trends_job(db)
    except Exception as e:
        db.rollback()
        raise
    finally:
        db.close()
```

`get_db()`가 하는 일을 직접 쓴 것. 동일한 패턴이다.

---

## job 단위 세션 vs 함수 단위 세션

**선택지 A: job 시작에서 세션 하나 (위 코드)**
```
장점: 코드 단순, 하나의 트랜잭션으로 묶임
단점: 중간에 에러 나면 전체 롤백, 세션 하나가 오래 열려 있음
```

**선택지 B: 함수마다 세션 열고 닫기**
```python
def run_news_job():
    db = SessionLocal()
    try:
        ...
        db.commit()
    finally:
        db.close()

def run_policy_job():
    db = SessionLocal()
    try:
        ...
        db.commit()
    finally:
        db.close()
```
```
장점: 각 단계가 독립적 (뉴스 저장 성공 후 정책에서 실패해도 뉴스는 보존)
단점: 코드 반복, 세션 여러 번 열고 닫음
```

**우리 프로젝트 판단:**
collector의 각 단계(뉴스/정책/트렌드)는 서로 독립적이라 실패가 전파될 이유 없음.
→ **선택지 B** (함수마다 세션) 또는 각 processor에 세션 직접 주입이 적합.

---

## 정리

```
get_db() = SessionLocal() + yield + finally close()
         = 세션 생성 → 라우터에 전달 → 응답 후 반납

collector = get_db() 안의 로직을 직접 재현
          = SessionLocal() + try/finally + close()
```
